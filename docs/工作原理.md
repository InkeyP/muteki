# 無敵 · 工作原理 (How it works)

無敵让一群异构的编码 Agent（Claude Code / Codex / cursor-agent）扑同一道题，在一张**共享黑板**上协作：谁发现的事实大家都能用，谁走过的死路大家都不再试，而 flag 只有**逐字出现在真实执行输出里**才被接受。核心不是「换个更强的脑子」，而是 **异构 + 共享证据 + 溯源闸门**。

而 worker 是怎么把数据交到平台、又怎么看到队友进展的？**全靠每个 worker 内置的 `muteki-blackboard` skill**——这是 worker 与黑板之间唯一的数据通道。

---

## 一张图看懂：解题阶段 × agent 循环

外层 `①②③④` 是一次 run 的四个阶段，内层 `(1)~(5)` 是阶段 ③ 每一拍的协作循环。难题的功夫全在 ③ 这个圈里，而圈里 worker ↔ 黑板的每一次读写都走 `muteki-blackboard` skill。

```
① 准备 Prepare
   建黑板 · 暂存题目附件 · 探活各引擎 ·（容器模式）起容器 + 反向连接
   每个 worker 引擎都装好 muteki-blackboard skill（容器镜像已内置）
        │
        ▼  仅「冷启动」才走侦察
② 侦察 Race ──── 每个健康引擎放 1 个单发 worker，并行扑整题（开局加速）
   ├─ 有人拿到 flag ───────────────────────────► 直接跳 ④（快路径 · 秒解）
   └─ 没人拿到 → 他们的 fact 已落黑板，协调器带着热数据接管
        │
        ▼
③ 协调主循环 Coordinator ── OODA，每 2s 一拍，循环到解出 / 被停
 ─────────────────────────────────────────────────────────────────────
   协调器 ──(1) 读整块黑板──► Reason 规划器（deepseek）
     ▲                                  │ (2) 提出 ≤4 个「下一步」intent
     │ 读图来规划                         ▼
     │   ┌──────────────── 共享黑板 SharedGraph ────────────────┐
     └───┤   Facts   Intents   Dead-ends   Flags   PoCs        │
         └──────▲──────────────────────────────▼───────────────┘
          写板 │（flag 必经闸门 Gate 校验）         │ 读板
         ┌──────┴──────────────────────────────┴───────────────┐
         │   ★ muteki-blackboard skill ── blackboard.py  ★      │  ← 核心数据通道
         │   写: write-fact / mark-deadend / claim / claim-resource
         │   读: read-facts / read-deadends / list-intents / ...
         └──────▲──────────────────────────────▼───────────────┘
      (5) 写回 │ fact/flag                      │ (3) 认领 intent / 读已知
         ┌──────┴──────────────────────────────┴───────────────┐
         │   WorkerA(claude)   WorkerB(codex)   WorkerC(cursor) │
         │   (4) 各自跑真实命令 / 工具，拿到真实输出 ── 单发 · 异构  │
         └──────────────────────────────────────────────────────┘
                          每写一条 → InsightBus 内存广播给在跑的队友
   ┄ 并行旁路 Review：质疑可疑 fact、压制重复路线（不挡主线）
   ┄ 全程 Guard：预算 / 无产出软暂停 / 操作员命令 / worker 举手
 ─────────────────────────────────────────────────────────────────────
        │  攒够合规 flag / 操作员叫停 / 预算耗尽
        ▼
④ 收尾 Finalize
   落 winner · 释放所有 intent 认领 · 发出唯一的 RUN_FINISHED · 清扫临时目录
```

**`(1)→(5)` 一圈就是无敵的核心**：协调器读黑板 → Reason 规划下一步 → intent 上黑板 → worker 各认一个跑真实命令 → **经 skill 把结果写回黑板（flag 还要过闸门）**，然后再读……每 2 秒转一圈，难题就是这样一圈圈把证据攒厚的。外层 `①②③④` 则是一次 run 的完整时间线。

> 注意：**协调器/Reason 在宿主进程内直接读写黑板**；而 **worker 是独立 CLI 进程**（容器模式还在另一个容器里），它读写黑板**只能走 `blackboard.py` skill**。

| 阶段 | 什么时候进 | 干什么 | 产出 |
|---|---|---|---|
| **① 准备** | run 一开始 | 建黑板、暂存附件、探活引擎、装好 skill、（容器模式）起容器+反向连接 | 空黑板 + 可用引擎 + 接好通道 |
| **② 侦察 Race** | 仅冷启动（复盘已解的题跳过） | 多引擎并行单发扑整题，做广度侦察 | flag（→快路径）或一批 fact |
| **③ 协调主循环** | 侦察没直接解出时 | `(1)~(5)` 不断转圈，随证据扩张 swarm | 黑板持续长大，直到攒够 flag |
| **④ 收尾** | 攒够 flag / 操作员停 / 预算耗尽 | 落 winner、释放认领、发终态事件、清扫 | RUN_FINISHED + 可复盘的黑板 |

**几个关键点：**
- **②→④ 的快路径**：简单题在侦察阶段就被秒了，根本不进主循环——这是無敵「简单题快、难题深」的来源。
- **③ 是循环不是一遍**：每 2 秒一拍，但**只有黑板变化了才重新规划**，不空转烧钱。
- **③ 里 Review 是并行旁路**，不挡主线，专门防脏数据把黑板带偏。
- **④ 幂等且必达**：正常解出、操作员叫停、还是崩溃，都会走到收尾，发出**唯一**终态事件让前端落定。
- 每次阶段切换（如 `race → coordinator`）协调器都会在事件流里发个标记，指挥台能实时显示跑到哪一阶段。

## 核心数据通道：muteki-blackboard skill

worker 是独立的 CLI Agent 进程（容器模式下还在另一个容器里），**够不到协调器的内存**。它能把数据交到平台、也能看到队友的进展，**全靠 `muteki-blackboard` 这个 skill**——也就是每个 worker 引擎里都装好的命令行脚本 `blackboard.py`。这是 worker 与黑板之间**唯一**的数据通道，也是「群体协作」真正落地的地方。

- **接入方式**：协调器给每个 worker 注入三个环境变量——`MUTEKI_BLACKBOARD_DB`（指向这道题的共享 SQLite）、`MUTEKI_WORKER_ID`（它是谁）、`MUTEKI_INTENT_ID`（它在做哪个 intent）。skill 本体在容器镜像里已内置（源码模式下直接用仓库里的那份），CLI Agent 会自动发现并按需调用。
- **它让 agent 在对的时机做对的事**（纪律写死在 `SKILL.md` 里）：
  - **动手前先读板**：`read-deadends` / `read-review`（别走已被否证或被 Review 压制的路）、`read-facts` / `read-routes` / `read-branches`（捡队友验证过的踏脚石）、`read-flags`（多 flag 题去补缺的那个）、`read-directives`（看操作员指令）。
  - **领活**：`list-intents` → `claim I3`（打印 `WON`/`LOST`，输了就换一个，天然防撞车）。
  - **危险/独占动作前锁资源**：`claim-resource "destructive:tcp:445@host"` / `release-resource`（两个 worker 不会同时打同一个靶 / 端口 / 账号）。
  - **确认了就写回**：`write-fact "..." --verified`（只有真在输出里看到的才加 `--verified`）、`mark-deadend "..."`（写死路和写事实一样有价值）。
- **为什么是 skill 而不是直接写库**：
  1. worker 跨进程 / 跨容器，CLI 脚本是它唯一能用的接口；
  2. `write-fact --verified` 仍要在宿主侧过**闸门 / 见证**校验——agent 自己说「verified」不等于真 verified；
  3. `SKILL.md` 把「何时读、何时写」的纪律直接教给 agent，让它**主动**协作，而不是被动被系统抓取。
- **兜底（不是主路）**：万一 worker 没调 skill，它在输出里打印的 `VERIFIED_FACT=` / `FOUND_FLAG=` 等标记也会被宿主收割进黑板——但那只是安全网，**主通道始终是 skill**。

## 把图里每块拆开讲

- **协调器 Coordinator**——宿主上的调度大脑，跑一个 2 秒一拍的 OODA 循环。**它自己从不解题**，只负责「读图、决定派什么、派几个、何时停」。它在进程内**直接**读写黑板。
- **Reason 规划器**——用一个便宜的推理模型（deepseek），把整块黑板喂进去，判断目标是否达成，没达成就吐出几个「下一步该探什么」的 **intent**。只有黑板**变化了**才触发规划，不是每拍都问。
- **共享黑板 SharedGraph**——每题一个文件、事件溯源的证据图，是**唯一事实来源**。它存五样东西：**Facts / Intents / Dead-ends / Flags / PoCs**。
- **muteki-blackboard skill**——每个 worker 内置的 `blackboard.py`，是 **worker 读 / 写黑板的唯一通道**：动手前读已知、领 intent、锁资源、确认后写 fact / dead-end。详见上一节。
- **Worker**——一个「带壳的 CLI Agent」，**单发**：领一个任务、跑到一个自然退出点就结束，不中途打断。它们用**不同引擎**，盲区不重叠，合在一起覆盖更多题型。所有与黑板的交互都经 skill。
- **闸门 Gate**——整个项目的护城河。一个 flag 要被接受，必须**格式对** + **不是占位符** + **逐字出现在真实命令输出里**。过不了就只当「候选」，模型无法把幻觉 flag 蒙混过关。
- **InsightBus**——黑板是持久存储（要落盘、要审计），InsightBus 是它旁边的**内存广播**：谁刚确认一条事实，立刻推给**正在跑**的队友，不用等他们去查数据库。

## Worker 的四种角色

所有 worker 都是**单发**的：领一个任务跑完就退，不中途 resume。操作员的指令、队友的发现，都只注入到**下一个**新 worker，而不去搅动正在跑的那个。

| 角色 | 干什么 |
|---|---|
| **race** | 开局，几个引擎并行扑整题做广度侦察，谁先拿 flag 谁赢 |
| **bootstrap** | 对整题做深度「冲刺」，一把梭试图直接解 |
| **explore** | 只认领**一个** intent，做窄而专的定向探测 |
| **review** | 审计员：挑战可疑事实、压制重复路线、拆分支；**自己不解题** |

> 没有「卡顿就杀 worker」的机制。worker 的回收只有两种：自然退出，或它认领的 intent **租约过期**被重新派发。

## 核心原语（各种术语的含义）

| 原语 | 含义 |
|---|---|
| **Fact（事实）** | 一条客观、已确认的发现。过了闸门叫 **verified**，没过只算 **candidate（候选，别当真）** |
| **Intent（意图）** | 一个声明出来、还没执行的探索方向。生命周期：**提出 → 认领 → 结论** |
| **Dead-end（死路）** | 一条已被否证的路线，记下来**让所有人别再走**——这是「群体不重复劳动」的关键 |
| **Flag** | 题目答案，只能由闸门接受；支持一题多 flag |
| **PoC** | 可被别的 worker 继承的利用产物（脚本 / payload） |
| **去重三件套** | **route**（同一攻击方向折叠）/ **lane**（危险独占资源串行化）/ **branch**（竞争假设分头证伪），防止 N 个 worker 撞车 |
| **Directive（操作员指令）** | 人的判断作为一等对象注入，agent 下次读板时吸收 |

## 调度：派什么 / 派几个 / 何时停

- **派什么**：Reason 从黑板的证据里提 intent；worker 用 skill `claim` 去认领。
- **派几个**：开局 2 个 bootstrap，普通 worker 上限 ~10；空槽每拍只补 1 个（避免一窝 worker 同生共死、把靶机限流打挂）；优先选**当前没在跑**的引擎（异构互补）。
- **何时停**：**第一个合规 flag 即胜**，其余 worker 立刻取消（多 flag 题则继续凑够数）。CTF 保证有解，所以 swarm **默认永不自己放弃**；只有「连续好几个 worker 毫无产出」时会**软暂停**等人，或操作员手动停。
- **健康降级**：派发前快速探活每个引擎，挂了的临时踢出名册，恢复了再归队。

## 事件架构

核心和前端之间**只有一个契约：一条有类型、有序、带单调序号的事件流**。

- **EventBus**：核心把每件事（worker 上下线、工具调用、证据进图、flag 找到、worker 举手…）发到一根进程内的 pub/sub 脊椎；带环形回放 + 落盘（每 run 一个 JSONL）。
- **前端是哑订阅者**：Web 指挥台和 TUI **从不直接调核心**，只做两件事——**订阅事件流**（SSE，断线用 Last-Event-ID 续传）和 **POST 命令**。所以两个前端完全可互换。
- **HITL 回路（人 → 核心）**：操作员的 `提示 / 改向 / 暂停 / 继续 / 停止 / 判假 flag / spawn / kill` 都是 POST 进一个队列，协调器在循环里消费。注意是**单发架构**——指令不会打断正在跑的 worker，而是注入**下一个**新 worker。
- **worker 举手（核心 → 人）**：worker 缺只有人能给的东西（VPS、凭证）或环境挂了，会发 `HITL_REQUEST`，协调器暂停这条方向等人，而不是傻乎乎拿更多 worker 去撞墙。

---

## 最外层（一句话带过）

最外面还有一层**运行后端**：`local` 直接在宿主跑 CLI（最快）；`container` 把 worker 关进 Kali 工具容器，worker **不开任何端口**，由容器内一个 Go supervisor **反向拨号**回宿主、宿主再把启动/信号命令发回去（一个 run 一条连接、按 worker 标识复用）。这层只搬运进程、保证工具链一致和按 run 隔离，**不参与任何解题判定**。

## 术语速查

| 术语 | 含义 |
|---|---|
| **muteki-blackboard skill** | 每个 worker 内置的 `blackboard.py`，是 worker 读/写黑板的**唯一通道**（读已知、领 intent、锁资源、写 fact/dead-end） |
| **single-shot（单发）** | worker 领一个任务跑完就退，不中途 resume；指令只影响下一个 worker |
| **race / bootstrap / explore / review** | 侦察 / 整题冲刺 / 单 intent 定向 / 审计，四种 worker 角色 |
| **coordinator** | 宿主上的 OODA 调度循环，自己不解题；进程内直接读写黑板 |
| **Reason 规划器** | deepseek，读黑板吐 intent；图变化时才触发 |
| **SharedGraph / 黑板** | 每题一个、事件溯源的证据图，唯一事实来源 |
| **InsightBus** | 黑板旁的内存广播，把新事实即时推给在跑的队友 |
| **Gate（闸门）** | 唯一硬编码、不可插拔的 flag 接受函数：格式 + 非占位符 + 逐字溯源 |
| **Fact / candidate / verified** | 客观发现 / 没过闸门的候选 / 过了闸门的已验证 |
| **Intent** | 待执行方向，提出→认领→结论 |
| **Dead-end** | 已否证的死路，防重试 |
| **route / lane / branch** | 方向去重 / 危险资源独占 / 竞争假设分支 |
| **Directive** | 一等的人工指令 |
| **first-valid-flag-wins** | 第一个过闸门的 flag 即胜，其余 worker 取消 |
| **HITL** | 人在环：人发命令 / worker 举手要资源 |
| **local / container 后端** | 宿主直跑 / Kali 容器反向连接隔离 |
